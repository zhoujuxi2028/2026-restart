## 🎯 工作流快速选择指南

### 我要开发新功能
```
场景：从零开始开发新功能
推荐：TDD工作流
步骤：先写测试 → 实现功能 → 重构优化
```

### 我遇到了Bug
```
场景：代码出现错误或测试失败
推荐：Debug SOP
步骤：复现问题 → 自主尝试15分钟 → 使用模板求助Claude
```

### 我要审查代码
```
场景：代码写完，需要检查质量
推荐：Code Review工作流
步骤：选择审查维度 → 使用模板审查 → 根据建议改进
```

### 我要完成一个完整项目
```
场景：需要从设计到部署的完整流程
推荐：完整开发循环
步骤：按6阶段逐步完成（设计→测试→实现→审查→重构→文档）
```

## 💡 工作流最佳实践

### ✅ DO（推荐做法）
1. **选择合适的工作流**：不要所有情况都用同一个
2. **严格遵循步骤**：不要跳过中间环节
3. **记录使用效果**：每次使用后评估
4. **持续优化**：根据经验调整流程

### ❌ DON'T（避免做法）
1. **不要过度流程化**：简单问题用简单方法
2. **不要生搬硬套**：根据实际情况灵活调整
3. **不要忽略文档**：每个流程都要留下记录
4. **不要孤立使用**：多个流程可以组合

## 🔧 工作流定制指南

### 何时定制
- 标准流程不适合你的场景
- 有更好的步骤顺序
- 需要添加项目特定步骤

### 如何定制
1. 从标准工作流开始
2. 记录不适应的地方
3. 调整步骤或添加新步骤
4. 测试新流程的效果
5. 文档化定制后的流程

### 定制示例
```
原始TDD流程：
1. 写测试
2. 实现功能
3. 重构

定制后（添加了Code Review）：
1. 写测试
2. 实现功能
3. Code Review
4. 重构
```
