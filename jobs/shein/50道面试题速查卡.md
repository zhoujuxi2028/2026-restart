## ⚡ K8s基础（8题）

### 1. 什么是Kubernetes？
```
✅ 开源容器编排平台
✅ 自动化部署、扩展和管理容器应用
✅ 解决：容器管理、自动扩缩容、高可用、滚动更新
```

### 2. Pod、Deployment、Service的区别？
```
Pod：
- 最小部署单元，包含1个或多个容器
- 临时的，会被重建

Deployment：
- 管理Pod副本数量
- 支持滚动更新和回滚
- 自动重启故障Pod

Service：
- 提供稳定的网络访问入口
- 负载均衡到多个Pod
- 类型：ClusterIP、NodePort、LoadBalancer
```

### 3. 为什么Pod需要Deployment管理？
```
✅ 副本管理（高可用）
✅ 滚动更新（零停机部署）
✅ 自动重启（自愈能力）
✅ 版本回滚
```

### 4. Service有哪些类型？
```
ClusterIP：集群内部访问（默认）
NodePort：暴露到节点端口，外部可访问
LoadBalancer：云平台负载均衡，需要云支持
```

### 5. K8s如何实现高可用？
```
✅ 多副本部署（replicas > 1）
✅ 自动重启故障Pod
✅ 滚动更新（逐步替换）
✅ HPA/CA自动扩缩容
✅ 健康检查（livenessProbe、readinessProbe）
```

### 6. 什么是滚动更新？
```
定义：逐步替换Pod，确保服务不中断

流程：
1. 创建新版本Pod
2. 等待新Pod就绪
3. 删除旧版本Pod
4. 重复，直到全部更新

优势：零停机部署
```

### 7. ConfigMap和Secret的区别？
```
ConfigMap：
- 存储非敏感配置
- 明文存储

Secret：
- 存储敏感数据（密码、证书）
- Base64编码（非加密，需配合RBAC）
```

### 8. Namespace的作用？
```
✅ 资源隔离（dev、test、prod）
✅ 权限控制（RBAC）
✅ 资源配额（ResourceQuota）
```

## ⚡ HPA/CA（6题）

### 9. HPA的工作原理？
```
HPA = Horizontal Pod Autoscaler

工作流程：
1. Metrics Server每15秒采集Pod指标（CPU/内存）
2. HPA Controller检查当前值 vs 目标值
3. 当前值 > 目标值 → 扩容（增加Pod）
4. 当前值 < 目标值 → 缩容（减少Pod，等待5分钟）
```

### 10. HPA如何触发扩容？
```
触发条件：CPU/内存使用率超过目标值

示例：
- 目标：CPU 50%
- 当前：CPU 80%
- 动作：扩容（增加Pod副本）

计算公式：
期望副本数 = 当前副本数 × (当前指标值 / 目标指标值)
```

### 11. CA的作用？
```
CA = Cluster Autoscaler

作用：集群资源不足时自动添加节点

触发条件：Pod因资源不足无法调度（Pending状态）
```

### 12. HPA和CA的区别？
```
┌──────────┬──────────────┬──────────────┐
│          │     HPA      │      CA      │
├──────────┼──────────────┼──────────────┤
│ 作用对象 │ Pod（应用层） │ Node（基础设施）│
│ 扩缩什么 │ Pod副本数    │ 集群节点数    │
│ 触发条件 │ CPU/内存高   │ Pod无法调度   │
│ 响应速度 │ 快（秒级）   │ 慢（分钟级）  │
│ 使用场景 │ 应用负载波动 │ 集群资源不足  │
└──────────┴──────────────┴──────────────┘
```

### 13. HPA扩缩容的延迟？
```
扩容：快（秒级）
- 检测周期：15秒
- 满足条件立即扩容

缩容：慢（分钟级）
- 稳定期：5分钟（避免频繁抖动）
- 指标持续低于目标值5分钟后才缩容
```

### 14. 如何测试HPA？
```
步骤：
1. 部署应用和HPA
2. 使用压测工具生成负载（K6、Locust）
3. 观察HPA：kubectl get hpa -w
4. 验证Pod数量变化：kubectl get pods -w
5. 检查扩缩容速度和稳定性

验证点：
✅ CPU升高时是否触发扩容
✅ 扩容速度是否合理
✅ 负载下降后是否缩容
✅ 缩容是否等待稳定期
```

```

❌ 资源不足（Insufficient cpu/memory）
❌ 配置错误（ConfigMap不存在）
❌ 健康检查失败
```

## ⚡ 混沌工程（10题）

### 21. 什么是混沌工程？
```
定义：通过主动注入故障验证系统稳定性的实践

核心思想：
✅ 不等待故障发生，主动制造故障
✅ 在生产环境验证系统韧性
✅ 提前发现系统薄弱点
```

### 22. 混沌工程的核心原则？
```
5大原则：
1️⃣ 定义稳态假设（系统正常时的指标）
2️⃣ 模拟真实世界事件（Pod崩溃、网络延迟）
3️⃣ 在生产环境运行实验
4️⃣ 自动化实验（持续验证）
5️⃣ 最小化爆炸半径（限制影响范围）
```

### 23. 常见的故障注入类型？
```
1. Pod故障：pod-kill、pod-failure
2. 网络故障：delay、loss、partition
3. 资源压力：CPU压力、内存压力
4. IO故障：IO延迟、IO错误
5. 时钟偏移：时间跳变
```

### 24. Chaos Mesh是什么？
```
定义：Kubernetes原生的混沌工程平台

特点：
✅ 云原生设计，用K8s资源管理
✅ 支持多种故障类型
✅ Web UI可视化操作
✅ 安全控制（Namespace隔离）
```

### 25. 如何测试系统的自愈能力？
```
方法：
1. 使用PodChaos杀死Pod
   apiVersion: chaos-mesh.org/v1alpha1
   kind: PodChaos
   spec:
     action: pod-kill
     mode: one

2. 观察Pod是否自动重启
   kubectl get pods -w

3. 验证流量是否自动切换
   持续请求Service，检查无中断

期望结果：
✅ Pod自动重启
✅ 服务无感知
✅ 流量自动切换到其他Pod
```

### 26. 如何注入网络延迟？
```
使用NetworkChaos：
apiVersion: chaos-mesh.org/v1alpha1
kind: NetworkChaos
metadata:
  name: network-delay-test
spec:
  action: delay
  mode: one
  selector:
    labelSelectors:
      app: my-app
  delay:
    latency: "100ms"    # 增加100ms延迟
    jitter: "10ms"      # ±10ms抖动
  duration: "1m"
```

### 27. 混沌实验失败了怎么办？
```
步骤：
1. 立即停止实验
   kubectl delete -f chaos.yaml

2. 分析失败原因
   - 查看Pod日志
   - 查看监控指标
   - 查看告警记录

3. 修复发现的问题
   - 增加资源限制
   - 优化超时设置
   - 增加重试机制

4. 小范围重试
   先在测试环境验证修复效果
```

### 28. 混沌工程和压力测试的区别？
```
混沌工程：
- 注入故障（Pod崩溃、网络延迟）
- 验证系统韧性
- 关注：容错能力

压力测试：
- 增加负载（大量请求）
- 验证性能极限
- 关注：吞吐量、延迟
```

### 29. 如何设计混沌实验？
```
步骤：
1. 定义稳态假设
   例：API响应时间 < 200ms，错误率 < 0.1%

2. 选择故障类型
   例：Pod崩溃、网络延迟

3. 小范围试验
   先在1个Pod上测试

4. 观察影响
   监控指标是否偏离稳态

5. 扩大范围
   逐步增加影响范围

6. 自动化
   定期自动运行实验
```

### 30. 混沌工程的价值？
```
✅ 提前发现系统薄弱点

✅ 减少生产故障
```

## ⚡ 告警与预案（5题）

### 31. 告警分级有哪些？
```
Critical（严重）：
- 需要立即处理
- 例：服务不可用、数据丢失

Warning（警告）：
- 需要关注，但不紧急
- 例：CPU使用率高、磁盘空间不足

Info（信息）：
- 通知性消息
- 例：部署完成、配置更新
```

### 32. 如何避免告警风暴？
```
方法：
1. 告警聚合：同一服务的告警合并
2. 告警抑制：主服务故障时抑制下游告警
3. 告警分组：按团队、服务分组
4. 告警静默：维护期间临时静默
5. 告警去重：相同告警只发送一次
```

### 33. 什么是告警疲劳？
```
定义：告警太多太频繁，导致忽视重要告警

原因：
❌ 告警阈值设置不合理
❌ 大量误报
❌ 告警太频繁

解决：
✅ 提高告警质量（减少误报）
✅ 告警分级（突出重要告警）
✅ 告警聚合（减少数量）
```

### 34. 预案包含哪些内容？
```
预案模板：
1. 故障类型：Pod持续重启
2. 触发条件：重启次数 > 5次/10分钟
3. 影响范围：服务部分不可用
4. 严重等级：Critical
5. 应急步骤：
   - 检查日志
   - 检查资源
   - 回滚版本
6. 恢复时间：< 15分钟
7. 后续优化：根因分析 + 故障复盘
```

### 35. 如何设计有效的预案？
```
原则：
✅ 覆盖常见故障类型
✅ 步骤清晰，可操作
✅ 责任明确，谁来处理
✅ 定期演练，验证有效性
✅ 持续更新，从故障中学习

示例故障类型：
- Pod崩溃
- HPA不工作
- Service无法访问
- 磁盘空间不足
- 数据库连接失败
```

## ⚡ 分布式系统（5题）

### 36. 分布式系统的主要挑战？
```
1. 网络不可靠
   - 延迟、丢包、分区

2. 部分失败
   - 某些节点故障，系统继续运行

3. 一致性问题
   - 数据同步延迟

4. 可观测性差
   - 日志、指标分散在多个节点
```

### 37. 如何测试分布式系统一致性？
```
方法：
1. 写入数据到主节点
   POST /api/data {"key": "value"}

2. 在多个副本节点验证数据
   GET /api/data from node1
   GET /api/data from node2
   GET /api/data from node3

3. 验证数据一致
   assert all nodes return same data

4. 测试延迟
   记录同步延迟时间
```

### 38. 什么是网络分区？
```
定义：部分节点网络中断，无法通信

示例：
集群有3个节点：A、B、C
网络分区后：
- A能与B通信
- A不能与C通信
- B不能与C通信

影响：
❌ 数据不一致
❌ 脑裂（多个节点认为自己是主节点）

测试：
使用NetworkChaos的partition类型
```

### 39. 如何测试跨集群故障切换？
```
步骤：
1. 部署多集群环境
   - 主集群（北京）
   - 备集群（上海）

2. 模拟主集群故障
   关闭主集群或断开网络

3. 验证流量切换
   - 流量是否自动切到备集群
   - DNS是否更新

4. 验证数据一致性
   - 数据是否同步到备集群

5. 验证恢复时间
   - RTO（恢复时间目标）
   - RPO（恢复点目标）
```

### 40. 分布式链路追踪的作用？
```
作用：跟踪请求在微服务间的调用路径

好处：
✅ 定位性能瓶颈（哪个服务慢）
✅ 分析依赖关系
✅ 排查故障（哪个服务出错）

工具：
- Jaeger
- Zipkin
- AWS X-Ray

示例：
用户请求 → API Gateway → Auth Service → User Service → Database
追踪可以看到每个环节的耗时
```
