**必学内容**：

- [ ] HPA配置示例（15分钟）
- [ ] HPA测试方法（10分钟）

**核心知识点**：
```yaml
# HPA示例
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50  # CPU超过50%触发扩容
```

**HPA工作流程**：
1. Metrics Server收集Pod的CPU/内存使用率
2. HPA Controller每15秒检查一次指标
3. 当前值 > 目标值 → 扩容（增加Pod）
4. 当前值 < 目标值 → 缩容（减少Pod）

**测试方法**：
```bash
# 1. 部署应用和HPA
kubectl apply -f deployment.yaml
kubectl apply -f hpa.yaml

# 2. 生成负载
kubectl run -it load-generator --rm --image=busybox -- /bin/sh
while true; do wget -q -O- http://my-app-service; done

# 3. 观察HPA
kubectl get hpa -w
kubectl top pods
```

```

工作原理：
1. Metrics Server采集Pod指标
```

#### 模块3：CA（Cluster Autoscaler）（60分钟）⭐⭐⭐⭐⭐
**必学内容**：
- [ ] CA工作原理（20分钟）
- [ ] HPA vs CA的区别（20分钟）
- [ ] CA测试场景（20分钟）

**核心知识点**：

**CA vs HPA对比表**：
| 特性 | HPA | CA |
|------|-----|-----|
| **作用对象** | Pod（应用层） | Node（基础设施层） |
| **扩缩什么** | 增减Pod副本数 | 增减集群节点 |
| **触发条件** | CPU/内存使用率 | Pod无法调度（资源不足） |
| **响应速度** | 快（秒级） | 慢（分钟级，需启动新节点） |
| **使用场景** | 应用负载波动 | 集群资源不足 |

**CA工作流程**：
```
场景：所有Node资源已满，新Pod无法调度

1. Pod处于Pending状态（资源不足）
2. CA检测到有Pod无法调度
3. CA向云平台请求新节点（AWS/GCP/阿里云）
4. 新Node加入集群
5. Scheduler将Pending的Pod调度到新Node
```

**HPA + CA协同工作**：
```
流程示例：
1. 流量增加 → CPU使用率上升
2. HPA触发 → 增加Pod副本（如从3个扩到10个）
3. 集群资源不足 → 部分Pod处于Pending
4. CA触发 → 添加新Node
5. Pod成功调度到新Node → 系统稳定
```

**测试方法**：
```bash
# 模拟CA场景
1. 创建大量Pod，耗尽集群资源
kubectl scale deployment my-app --replicas=100

2. 观察Pod状态
kubectl get pods -w
# 会看到部分Pod处于Pending

3. 检查CA日志
kubectl logs -n kube-system cluster-autoscaler-xxx

4. 验证新节点加入
kubectl get nodes -w
```

**面试话术**：
```
"CA和HPA是不同层次的自动扩缩：
- HPA：应用层，增减Pod副本，秒级响应
- CA：基础设施层，增减Node节点，分钟级响应

协同工作场景：
1. 流量激增，HPA增加Pod
2. 节点资源不足，Pod无法调度
3. CA检测到Pending的Pod
4. CA自动添加新节点
5. Pod成功调度，系统恢复

测试重点：
- HPA扩容后资源不足的场景
- CA是否正确添加节点
- 节点添加后Pod能否正常调度
- 缩容时是否安全驱逐Pod"
```

**必学内容**：

**混沌工程定义**：
```
定义：通过主动注入故障来验证系统稳定性的实践

核心思想：
- 在生产环境（或类生产）中进行实验
- 主动制造故障（而非等待故障发生）
- 验证系统的容错能力
- 提前发现薄弱点

为什么需要混沌工程？
- 分布式系统复杂，故障不可避免
- 被动等待故障代价太高
- 需要主动验证系统韧性
```

**混沌工程原则**：
```
1. 定义稳态假设
   "系统正常时应该是什么样的？"
   例：API响应时间 < 200ms，错误率 < 0.1%

2. 模拟真实世界事件
   - Pod崩溃
   - 网络延迟/丢包
   - CPU/内存压力
   - 节点宕机

3. 在生产环境运行实验
   从小规模开始，逐步扩大

4. 自动化运行实验
   持续验证系统韧性

5. 最小化爆炸半径
   限制实验影响范围，快速回滚
```

**故障注入类型**：
```yaml
# Chaos Mesh支持的故障类型

1. PodChaos - Pod故障
   - pod-kill: 杀死Pod
   - pod-failure: Pod不可用
   - container-kill: 杀死容器

2. NetworkChaos - 网络故障
   - delay: 网络延迟
   - loss: 丢包
   - duplicate: 重复包
   - corrupt: 数据损坏
   - partition: 网络分区

3. StressChaos - 资源压力
   - CPU压力
   - 内存压力

4. IOChaos - IO故障
   - 延迟
   - 错误
   - 丢失数据

5. TimeChaos - 时钟偏移
   - 时间跳变
```

**Chaos Mesh示例**：
```yaml
# PodChaos示例：杀死Pod
apiVersion: chaos-mesh.org/v1alpha1
kind: PodChaos
metadata:
  name: pod-kill-test
spec:
  action: pod-kill
  mode: one           # 每次杀一个
  selector:
    namespaces:
      - default
    labelSelectors:
      app: my-app
  scheduler:
    cron: "@every 2m"  # 每2分钟执行一次
```

**面试话术**：
```
"混沌工程是通过主动注入故障验证系统稳定性的实践。

核心步骤：
1. 定义稳态假设（正常状态的指标）
2. 注入故障（Pod崩溃、网络延迟）
3. 观察系统是否偏离稳态
4. 修复发现的问题

常用工具：
```

**告警系统**：
```

告警流程：
1. Prometheus收集指标
2. 触发告警规则
3. Alertmanager分组、去重、路由
4. 发送通知（邮件、钉钉、PagerDuty）
```

**告警规则示例**：
```yaml
# Prometheus告警规则
groups:
- name: pod-alerts
  rules:
  - alert: PodMemoryHigh
    expr: container_memory_usage_bytes / container_spec_memory_limit_bytes > 0.9
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "Pod内存使用超过90%"
      description: "{{ $labels.pod }} 内存使用率: {{ $value }}"
```

**告警治理**：
```
常见问题：
1. 告警风暴：短时间大量告警
2. 告警疲劳：太多无效告警
3. 漏报：关键问题没告警

优化方法：
1. 告警分级：Critical、Warning、Info
2. 告警聚合：同一服务的告警合并
3. 告警抑制：依赖服务故障时抑制下游告警
4. 告警静默：维护期间临时静默
```

**预案设计**：
```
预案模板：

故障类型：Pod持续重启
触发条件：Pod重启次数 > 5次/10分钟
影响范围：服务不可用
应急步骤：
1. 检查Pod日志：kubectl logs <pod>
2. 检查资源限制：kubectl describe pod <pod>
3. 回滚到上一版本：kubectl rollout undo deployment <name>
4. 联系开发团队
恢复时间：< 15分钟
```

```
3. 告警聚合：避免告警风暴
4. 响应时效：关键告警 < 5分钟响应

预案设计：
- 每种故障类型都有预案
- 明确触发条件和应急步骤
- 定期演练和更新
- 记录故障复盘
```

22. Q: 混沌工程的核心原则？
    A: 定义稳态、模拟真实故障、生产环境实验、自动化、最小化影响

23. Q: 常见的故障注入类型？
    A: Pod故障、网络故障、资源压力、IO故障、时钟偏移

24. Q: Chaos Mesh是什么？
    A: Kubernetes原生的混沌工程平台

25. Q: 如何测试系统的自愈能力？
    A: 用PodChaos杀死Pod，观察是否自动重启

26. Q: 如何注入网络延迟？
    A: 用NetworkChaos的delay类型

27. Q: 混沌实验失败了怎么办？
    A: 立即停止实验，分析原因，修复问题后再试

28. Q: 混沌工程和压力测试的区别？
    A: 混沌工程注入故障，压力测试增加负载

29. Q: 如何设计混沌实验？
    A: 定义稳态 → 选择故障类型 → 小范围试验 → 扩大范围

30. Q: 混沌工程的价值？
    A: 提前发现系统薄弱点，提升韧性，减少生产故障

**告警与预案（5题）**
31. Q: 告警分级有哪些？
    A: Critical（需立即处理）、Warning（需关注）、Info（信息）

32. Q: 如何避免告警风暴？
    A: 告警聚合、抑制、分组

33. Q: 什么是告警疲劳？
    A: 告警太多太频繁，导致忽视重要告警

34. Q: 预案包含哪些内容？
    A: 故障类型、触发条件、影响范围、应急步骤、恢复时间

35. Q: 如何设计有效的预案？
    A: 覆盖常见故障、步骤清晰、定期演练、持续更新

**分布式系统（5题）**
36. Q: 分布式系统的主要挑战？
    A: 网络不可靠、部分失败、一致性、可观测性

37. Q: 如何测试分布式系统一致性？
    A: 数据写入后，在多个节点验证数据一致

38. Q: 什么是网络分区？
    A: 部分节点网络中断，无法通信

39. Q: 如何测试跨集群故障切换？
    A: 模拟主集群故障，验证流量是否切到备集群

40. Q: 分布式链路追踪的作用？
    A: 跟踪请求在微服务间的调用路径，定位性能瓶颈

**综合测试（10题）**
41. Q: 你如何保证云平台测试质量？
    A: 自动化测试 + 混沌工程 + 监控告警 + 回归测试

42. Q: 如何设计自动化测试框架？
    A: 模块化、可复用、清晰断言、持续集成

43. Q: 测试环境如何隔离？
    A: 使用Namespace、独立集群、数据隔离

44. Q: 如何测试滚动更新？
    A: 更新过程中持续请求，验证无中断

45. Q: 如何提升测试覆盖率？
    A: 单元测试 + 集成测试 + 端到端测试 + 混沌测试

46. Q: 如何处理测试环境不稳定？
    A: 定期重建环境、资源限制、自动清理

47. Q: 你用过哪些测试工具？
    A: Python、Pytest、Kubernetes库、Chaos Mesh、K6

48. Q: 如何定位复杂问题？
    A: 日志分析、指标监控、分布式追踪、故障复现

49. Q: CI/CD中如何集成测试？
    A: 代码提交 → 单元测试 → 构建镜像 → 集成测试 → 部署

50. Q: 你最擅长的测试领域？
    A: 自动化测试、云平台测试、稳定性测试
