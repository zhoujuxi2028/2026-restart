# Day 7: English Communication Templates

## Technical Vocabulary

### Pronunciation Guide
- **STLC**: ess-tee-ell-see (Software Testing Life Cycle)
- **ISTQB**: eye-ess-tee-kew-bee (International Software Testing Qualifications Board)
- **BDD**: bee-dee-dee (Behavior-Driven Development)
- **Gherkin**: gur-kin (BDD syntax language)
- **Equivalence Partitioning**: ee-kwiv-uh-lense par-tish-un-ing
- **Boundary Value Analysis**: bown-dree val-yoo uh-nal-uh-sis

### Key Terms
- **Three Amigos**: Dev, QA, Business collaboration
- **Given-When-Then**: BDD scenario structure
- **Page Object Model (POM)**: Design pattern for test automation
- **Traceability Matrix (RTM)**: Requirements ↔ Tests mapping
- **Shift-Left**: Start testing early in lifecycle
- **Test Pyramid**: Most unit tests, few E2E tests
- **Defect Density**: Defects per 1000 lines of code
- **Living Documentation**: Tests as up-to-date docs

## Self-Introduction Extension

"I have strong knowledge of testing methodologies. I follow STLC systematically, use BDD for collaboration, and apply ISTQB test design techniques like equivalence partitioning and boundary value analysis. I design test frameworks using Page Object Model, ensuring maintainability and scalability. I use risk-based prioritization and the test pyramid to balance testing efforts."

## STAR Stories

### Story 1: Implementing BDD

**Situation**: "Miscommunication between business, dev, and QA about requirements caused rework."

**Task**: "Improve collaboration and ensure shared understanding before development."

**Action**: "Introduced BDD with Three Amigos sessions. Business, dev, and QA wrote Gherkin scenarios together. Automated scenarios using Cucumber and Cypress."

**Result**: "Rework reduced 40%. Business could read tests. Discovered 20+ edge cases upfront. Developer confidence improved."

### Story 2: Test Automation Framework Design

**Situation**: "Multiple teams had inconsistent automation with duplicated code and poor maintainability."

**Task**: "Design unified enterprise test automation framework."

**Action**: "Designed layered architecture with Page Object Model, implemented design patterns (Factory, Singleton), created clear structure and documentation."

**Result**: "Framework adopted by 5 teams (25 developers). Onboarding reduced from 1 week to 1 day. Maintenance time decreased 70%. Scaled from 100 to 500+ tests."

## Common Phrases

### Explaining STLC
- "STLC provides **a systematic approach** to testing"
- "Each phase has **clear entry and exit criteria**"
- "**Shift-left testing** means starting early"

### Discussing BDD
- "BDD **facilitates collaboration** between stakeholders"
- "Gherkin scenarios serve as **executable documentation**"
- "Three Amigos **ensure shared understanding**"

### Describing POM
- "POM **separates test logic from page structure**"
- "Locators are **centralized** in page objects"
- "Tests **read like plain English**"

### Test Strategy
- "I use **risk-based prioritization** for high-impact areas"
- "I follow the **test pyramid**: most tests at unit level"
- "I **balance automation with manual**, typically 80/20"

## Response Template

### Explaining a Methodology
1. **Definition** (1 sentence): What is it?
2. **Purpose** (1 sentence): Why use it?
3. **Key Components** (2-3 points): What are the parts?
4. **Example** (1-2 sentences): Real application
5. **Benefits** (1-2 points): Why it matters

**Example:**
"STLC is the Software Testing Life Cycle. Its purpose is systematic, comprehensive testing. Six phases: requirement analysis, test planning, test case development, environment setup, test execution, test closure. Following STLC, we identified 15 requirement issues before development. This ensures nothing is missed."
