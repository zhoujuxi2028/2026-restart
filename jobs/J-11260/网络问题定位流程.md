# 网络问题定位流程与工具清单
> 系统化网络故障排查方法论

## 🎯 网络故障排查基本原则

### DOWNTIME故障分析模型
```
D - Define (定义问题)
O - Outline (概述症状)
W - Why (为什么发生)
N - Now (现在状态)
T - Timeline (时间线)
I - Impact (影响范围)
M - Method (解决方法)
E - Evaluate (评估结果)
```

### 故障排查黄金法则
1. **先易后难**: 从简单检查开始
2. **自下而上**: 从物理层向应用层排查
3. **分而治之**: 逐步缩小问题范围
4. **记录一切**: 详细记录排查过程
5. **假设验证**: 建立假设并逐一验证

## 🔍 分层故障排查流程

### OSI七层排查法

**Layer 1 - 物理层**
```bash
检查项目：
□ 网线连接状态
□ 端口指示灯状态
□ 设备电源状态
□ 光纤连接(如有)

故障现象：
- 完全无连接
- 间歇性连接中断
- 端口无指示灯

诊断命令：
ethtool eth0                    # 查看网卡状态
dmesg | grep -i network         # 查看网络相关内核消息
lspci | grep -i network         # 查看网络硬件
ip link show                    # 查看接口状态

故障案例：
现象: 服务器无网络连接
排查: ethtool eth0显示"Link detected: no"
原因: 网线松动
解决: 重新插接网线
```

**Layer 2 - 数据链路层**
```bash
检查项目：
□ MAC地址表
□ VLAN配置
□ 交换机端口配置
□ 生成树协议状态

故障现象：
- 同网段无法通信
- VLAN间无法访问
- 交换环路导致网络风暴

诊断命令：
arp -a                          # 查看ARP表
ip neighbor show                # 显示邻居表
bridge fdb show                 # 查看MAC地址表
vconfig                         # VLAN配置查看

交换机诊断：
show mac-address-table          # 显示MAC地址表
show vlan                       # 显示VLAN配置
show spanning-tree              # 显示STP状态

故障案例：
现象: 主机A无法ping通主机B(同网段)
排查: arp -a显示主机B的MAC地址为incomplete
原因: ARP解析失败，可能是交换机端口问题
解决: 检查交换机端口配置，重启端口
```

**Layer 3 - 网络层**
```bash
检查项目：
□ IP地址配置
□ 子网掩码
□ 默认网关
□ 路由表

故障现象：
- 跨网段无法通信
- 路由不可达
- IP地址冲突

诊断命令：
ip addr show                    # 查看IP配置
ip route show                   # 查看路由表
ping -c 4 <gateway>             # 测试网关连通性
traceroute <destination>        # 追踪路由路径
netstat -rn                     # 显示路由表(传统命令)

路由器诊断：
show ip route                   # 显示路由表
show ip interface brief        # 显示接口IP状态
ping <destination>              # 路由器ping测试

故障案例：
现象: 无法访问外网
排查: ping网关成功，ping外网DNS失败
原因: 路由器到ISP链路故障
解决: 联系ISP检查线路
```

**Layer 4 - 传输层**
```bash
检查项目：
□ TCP/UDP端口状态
□ 连接建立过程
□ 防火墙规则
□ 服务监听状态

故障现象：
- 服务无法连接
- 连接超时
- 端口被拒绝

诊断命令：
netstat -tulpn                  # 查看监听端口
ss -tulpn                       # 现代版netstat
lsof -i :80                     # 查看指定端口使用情况
telnet <host> <port>            # 测试端口连通性
nmap -sT <host>                 # 端口扫描

防火墙检查：
iptables -L -n                  # 查看iptables规则
ufw status                      # 查看ufw状态
systemctl status firewalld     # 查看firewalld状态

故障案例：
现象: Web服务无法访问
排查: netstat显示nginx监听80端口，telnet localhost 80成功
分析: 服务正常，检查防火墙和客户端网络
解决: 开放防火墙80端口
```

## 🛠 网络诊断工具箱

### 基础连通性测试
```bash
# 1. ping - 基础连通性测试
ping -c 4 192.168.1.1           # 基础ping测试
ping -f 192.168.1.1             # 洪水ping(压力测试)
ping -s 1472 192.168.1.1        # 大包ping测试
ping6 2001:db8::1               # IPv6 ping测试

# 2. traceroute - 路径追踪
traceroute google.com           # 追踪到目标的路由
traceroute -I google.com        # 使用ICMP追踪
mtr google.com                  # 持续追踪(更强大)

# 3. nslookup/dig - DNS诊断
nslookup google.com             # 基础DNS查询
dig google.com                  # 详细DNS查询
dig @8.8.8.8 google.com         # 指定DNS服务器查询
dig google.com MX               # 查询邮件记录
```

### 高级网络分析工具
```bash
# 1. tcpdump - 命令行抓包
tcpdump -i eth0 host 192.168.1.100
tcpdump -i any port 80
tcpdump -i eth0 -w capture.pcap

# 2. nmap - 网络扫描
nmap -sn 192.168.1.0/24         # 主机发现扫描
nmap -sT 192.168.1.100          # TCP连接扫描
nmap -sU 192.168.1.100          # UDP扫描
nmap -O 192.168.1.100           # 操作系统检测

# 3. nc (netcat) - 网络瑞士军刀
nc -l 1234                      # 监听1234端口
nc 192.168.1.100 22             # 连接到SSH端口
nc -u 192.168.1.100 53          # UDP连接到DNS

# 4. iperf3 - 带宽测试
iperf3 -s                       # 服务器模式
iperf3 -c 192.168.1.100         # 客户端连接测试
iperf3 -c 192.168.1.100 -u      # UDP测试
```

### 系统网络状态监控
```bash
# 1. netstat - 网络连接状态
netstat -tulpn                  # 监听端口
netstat -an | grep ESTABLISHED  # 已建立连接
netstat -i                      # 接口统计
netstat -s                      # 协议统计

# 2. ss - 现代网络统计工具
ss -tulpn                       # 监听端口(更快)
ss -o state established         # 已建立的连接
ss -i                           # 详细接口信息

# 3. iftop - 实时流量监控
iftop -i eth0                   # 监控指定接口
iftop -n                        # 不解析主机名

# 4. vnstat - 流量统计
vnstat -i eth0                  # 接口流量统计
vnstat -d                       # 按天统计
vnstat -m                       # 按月统计
```

## 📋 系统化故障排查流程

### 1. 信息收集阶段
```bash
故障描述收集清单：
□ 故障现象详细描述
□ 故障发生时间
□ 影响范围(用户、服务)
□ 最近的变更操作
□ 错误信息截图
□ 网络拓扑图

环境信息收集：
hostname                        # 主机名
uname -a                        # 系统信息
uptime                          # 运行时间
who                            # 当前登录用户
last | head                     # 最近登录记录

网络配置收集：
ip addr show                    # IP配置
ip route show                   # 路由配置
cat /etc/resolv.conf            # DNS配置
cat /etc/hosts                  # 本地解析
```

### 2. 基础连通性验证
```bash
# Step 1: 本机网络配置检查
ping 127.0.0.1                 # 回环测试
ping $(ip route | grep default | awk '{print $3}')  # 网关测试

# Step 2: 内网连通性测试
ping 192.168.1.1               # 网关ping
ping 192.168.1.100             # 内网其他主机

# Step 3: 外网连通性测试
ping 8.8.8.8                   # 公网DNS
ping google.com                # 域名解析测试

# Step 4: 服务端口测试
telnet 192.168.1.100 22        # SSH端口
telnet google.com 80           # HTTP端口
```

### 3. 深度分析阶段
```bash
# 路由分析
traceroute -n google.com        # 数值显示，避免DNS延迟
mtr -r -c 10 google.com         # 生成报告

# 端口分析
nmap -sT -O 192.168.1.100      # TCP扫描+OS检测
nc -zv 192.168.1.100 1-1024    # 端口连通性扫描

# 性能分析
iperf3 -c 192.168.1.100 -t 30  # 30秒带宽测试
ping -f -c 1000 192.168.1.100  # 延迟稳定性测试

# 抓包分析
tcpdump -i eth0 -w problem.pcap host 192.168.1.100
# 然后使用Wireshark分析pcap文件
```

## 🚨 常见故障场景与解决

### 场景1: 网站访问慢
**故障现象**: 用户反映网站访问速度慢

**排查流程**:
```bash
# 1. DNS解析测试
time nslookup www.example.com
dig www.example.com             # 查看DNS响应时间

# 2. 网络延迟测试
ping -c 10 www.example.com      # 基础延迟
mtr www.example.com             # 路径分析

# 3. HTTP响应测试
curl -w "@curl-format.txt" -o /dev/null -s "http://www.example.com/"
# curl-format.txt内容:
#      time_namelookup:  %{time_namelookup}\n
#         time_connect:  %{time_connect}\n
#      time_appconnect:  %{time_appconnect}\n
#     time_pretransfer:  %{time_pretransfer}\n
#        time_redirect:  %{time_redirect}\n
#   time_starttransfer:  %{time_starttransfer}\n
#                      ----------\n
#           time_total:  %{time_total}\n

# 4. 带宽测试
iperf3 -c speedtest.server.com

可能原因与解决：
- DNS解析慢 → 更换DNS服务器(8.8.8.8, 1.1.1.1)
- 网络延迟高 → 检查网络路径，联系ISP
- 服务器响应慢 → 检查服务器负载
- 带宽不足 → 升级网络带宽
```

### 场景2: 间歇性网络中断
**故障现象**: 网络连接时断时续

**排查流程**:
```bash
# 1. 持续监控连通性
ping -i 0.5 192.168.1.1 | tee ping.log  # 每0.5秒ping一次

# 2. 网络接口监控
watch -n 1 'ip link show eth0'         # 监控接口状态

# 3. 系统日志检查
tail -f /var/log/messages | grep -i network
journalctl -f -u NetworkManager        # systemd系统

# 4. 硬件状态检查
ethtool eth0                           # 网卡状态
dmesg | grep -i error                  # 硬件错误

可能原因与解决：
- 网线接触不良 → 更换网线
- 网卡驱动问题 → 更新驱动程序
- 交换机端口问题 → 更换端口
- 电磁干扰 → 改善布线环境
```

### 场景3: 特定服务无法访问
**故障现象**: SSH/HTTP等特定服务无法连接

**排查流程**:
```bash
# 1. 服务状态检查
systemctl status sshd              # 检查SSH服务状态
netstat -tulpn | grep :22          # 检查SSH端口监听

# 2. 防火墙规则检查
iptables -L -n | grep 22           # 检查iptables规则
ufw status                         # Ubuntu防火墙状态

# 3. SELinux检查(RHEL/CentOS)
getenforce                         # SELinux状态
audit2why < /var/log/audit/audit.log | grep sshd

# 4. 客户端测试
telnet server.ip 22                # 端口连通性
ssh -v user@server.ip              # SSH详细连接信息

可能原因与解决：
- 服务未启动 → systemctl start sshd
- 端口被占用 → 查找占用进程并处理
- 防火墙阻拦 → 开放相应端口
- 配置文件错误 → 检查/etc/ssh/sshd_config
```

## 🔧 电信设备特有排查

### GPON网络故障排查
```bash
# 1. OLT侧检查
show gpon olt 0                    # 查看OLT状态
show gpon onu state all            # 查看所有ONU状态
show gpon onu-optical-info all     # 查看光功率信息

# 2. 光功率检查
show interface gpon 0/1 optical-info    # 端口光功率
# 正常范围：发送功率 +2~+7dBm，接收功率 -8~-28dBm

# 3. OMCI通信检查
show gpon onu omci-info            # OMCI通信状态
show gpon onu capability           # ONU能力查询

故障判断标准：
- 光功率过低(<-28dBm) → 检查光纤连接
- 光功率过高(>+7dBm) → 检查光衰减
- OMCI通信失败 → ONU配置或硬件问题
```

### SNMP设备监控
```bash
# 1. SNMP连通性测试
```

## 📊 性能问题定位

### 带宽瓶颈分析
```bash
# 1. 实时流量监控
iftop -i eth0                      # 实时连接流量
nethogs                           # 按进程显示流量
vnstat -l -i eth0                 # 实时统计

# 2. 历史流量分析
vnstat -d -i eth0                 # 按天统计
sar -n DEV 1 10                   # 网络接口统计

# 3. 带宽测试
iperf3 -c target.server.com -t 60  # 60秒带宽测试
iperf3 -c target.server.com -P 4   # 4个并行连接

性能基准参考：
- 千兆网卡理论值: 1000Mbps
- 实际可用带宽: 通常90-95%
- 延迟标准: 局域网<1ms, 互联网<50ms
```

### 延迟问题分析
```bash
# 1. 基础延迟测试
ping -c 100 target.host | tail -1  # 100次ping统计
ping -f target.host                # 洪水ping

# 2. 路径分析
mtr -r -c 100 target.host          # 详细路径分析
traceroute -I target.host          # ICMP追踪

# 3. 应用层延迟
curl -w "%{time_total}" -o /dev/null target.url
httping -c 10 http://target.host   # HTTP ping

延迟问题定位：
- 物理层: 检查线路质量
- 数据链路层: 交换机处理延迟
- 网络层: 路由跳数过多
- 应用层: 服务器处理时间
```

### 文档记录模板
```
故障报告编号: INC-2026-0204-001
报告时间: 2026-02-04 10:30
报告人: 用户姓名

【故障描述】
- 现象: 无法访问内网服务器192.168.1.100
- 发生时间: 2026-02-04 09:45
- 影响范围: 开发部门10台主机
- 错误信息: "连接超时"

【排查过程】
1. 网络连通性检查
   - ping 192.168.1.100 → 超时
   - ping 192.168.1.1 → 正常

2. 路由分析
   - traceroute显示在第3跳中断
   - 确定问题在核心交换机

3. 设备检查
   - 核心交换机端口状态异常
   - 重启端口后恢复正常

【根本原因】
核心交换机端口硬件故障

【解决方案】
1. 立即解决: 重启故障端口
2. 长期方案: 更换交换机端口模块

【预防措施】
1. 增加端口监控告警
2. 建立备用链路
3. 定期硬件巡检
```

### 应急处理流程
```
紧急故障处理SOP：

1. 故障确认(5分钟内)
   □ 确认故障现象
   □ 评估影响范围
   □ 通知相关人员

2. 快速恢复(15分钟内)
   □ 尝试重启服务
   □ 切换备用线路
   □ 临时解决方案

3. 根因分析(30分钟内)
   □ 详细排查
   □ 确定根本原因
   □ 制定永久解决方案

4. 事后总结(24小时内)
   □ 故障报告
   □ 改进措施
   □ 知识库更新
```
